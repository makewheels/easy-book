# Easy Book - 后端架构设计文档

## 项目架构概览

Easy Book后端采用分层架构设计，主要包括API层、服务层、数据层和基础设施层。系统基于FastAPI框架构建，提供RESTful API接口。

## 服务层详细设计

### 1. 工具函数模块 (`services/utils.py`)

**职责**：提供通用的工具函数，被其他服务模块共享使用

**核心功能**：
- `calculate_dynamic_status_new()` - 基于新时间格式的动态状态计算
- `calculate_dynamic_status()` - 兼容旧格式的动态状态计算
- `convert_legacy_time_data()` - 时间数据格式转换工具

**代码行数**：约4,500字节

### 2. 学员服务模块 (`services/student.py`)

**职责**：处理学员相关的所有业务逻辑

**核心功能**：
- 学员创建、查询、更新、删除 (CRUD)
- 课程次数管理和扣费逻辑
- 利润计算（价格 - 场地分成）
- 学员数据预处理和验证

**主要方法**：
```python
class StudentService:
    async def create(student_data: dict) -> StudentModel      # 创建学员
    async def get_all(skip, limit) -> List[StudentModel]      # 获取学员列表
    async def get_by_id(student_id: str) -> StudentModel      # 根据ID查询
    async def update(student_id, data) -> StudentModel        # 更新学员信息
    async def delete(student_id: str) -> bool                 # 删除学员
    async def update_lessons(student_id, lessons) -> bool      # 更新课程次数
```

**代码行数**：约4,000字节

### 3. 预约服务模块 (`services/appointment.py`)

**职责**：处理预约相关的所有业务逻辑，是最复杂的业务模块

**核心功能**：
- 预约创建和冲突检查（支持新旧时间格式）
- 预约状态管理（scheduled, active, completed, cancel）
- 时间冲突检测（新格式：时间区间重叠，旧格式：相同时段）
- 课程次数扣减和恢复
- 预约查询和统计

**主要方法**：
```python
class AppointmentService:
    async def create(appointment_data: dict) -> AppointmentModel        # 创建预约
    async def check_time_conflict(student_id, start_time, end_time)     # 时间冲突检查
    async def check_conflict(student_id, date, time_slot)                # 格式冲突检查
    async def get_by_id(appointment_id: str) -> AppointmentModel          # 查询预约
    async def cancel(appointment_id: str) -> bool                         # 取消预约
    async def get_student_appointments(student_id, status) -> List[]       # 学员预约列表
    async def get_upcoming_appointments(days: int) -> List[dict]          # 即将到期预约
    async def get_week_appointments(start_date: date) -> dict            # 周预约数据
    async def get_daily_appointments(appointment_date: str) -> List[]     # 日预约数据
```

**代码行数**：约10,800字节（最大模块）

### 4. 考勤服务模块 (`services/attendance.py`)

**职责**：处理学员签到和考勤记录管理

**核心功能**：
- 学员签到处理（扣减课程次数）
- 预约取消标记（不扣减课程次数）
- 考勤记录查询和管理
- 支持新旧时间格式的考勤处理

**主要方法**：
```python
class AttendanceService:
    async def checkin(appointment_id: str, student_id: str) -> AttendanceModel  # 学员签到
    async def mark_cancel(appointment_id: str, student_id: str) -> AttendanceModel # 标记取消
    async def get_by_student(student_id: str) -> List[AttendanceModel]           # 学员考勤记录
```

**代码行数**：约6,500字节

### 5. 统一导出模块 (`services/__init__.py`)

**职责**：提供统一的导出接口，简化API层的导入操作

**导出内容**：
```python
# 工具函数
from .utils import (
    calculate_dynamic_status_new,
    calculate_dynamic_status,
    convert_legacy_time_data
)

# 服务类
from .student import StudentService
from .appointment import AppointmentService
from .attendance import AttendanceService
```

## 模块化优势

### 1. 代码组织清晰
- **按业务模块分离**：每个模块专注一个业务领域
- **职责单一**：每个文件职责明确，便于理解和维护
- **易于扩展**：新功能可以独立添加到对应模块

### 2. 团队协作友好
- **并行开发**：不同开发者可以同时开发不同模块
- **减少冲突**：代码变更影响范围小
- **代码审查**：模块边界清晰，审查更高效

### 3. 测试友好
- **单元测试**：每个模块可以独立测试
- **模拟依赖**：通过接口模拟其他模块
- **集成测试**：模块间组合测试更清晰

### 4. 维护性提升
- **bug定位**：问题可以快速定位到具体模块
- **功能修改**：局部修改，影响范围可控
- **代码重构**：模块内部重构不影响其他模块

## 向后兼容性

为了确保系统升级的平滑性，新的服务模块设计充分考虑了向后兼容性：

### 1. 统一导出接口
- API层代码无需修改，导入路径保持不变
- 通过 `__init__.py` 提供统一的导出接口

### 2. 数据格式兼容
- 同时支持新旧时间格式（start_time/end_time 和 appointment_date/time_slot）
- 自动格式转换，保证数据一致性

### 3. API接口不变
- 所有现有的API接口保持功能不变
- 新增功能通过新接口提供，不影响现有功能

## 性能考虑

### 1. 模块加载优化
- 按需导入，减少不必要的依赖加载
- 延迟加载，提高应用启动速度

### 2. 数据库操作优化
- 批量操作支持，减少数据库连接次数
- 索引优化，提高查询性能

### 3. 内存使用优化
- 避免循环导入，减少内存占用
- 合理的对象生命周期管理

## 未来扩展方向

### 1. 微服务化准备
- 当前模块化设计为未来微服务拆分打下基础
- 每个服务模块可以独立部署和扩展

### 2. 缓存层集成
- 为高频查询添加Redis缓存支持
- 提高系统响应速度

### 3. 事件驱动架构
- 引入事件总线，实现模块间的松耦合
- 支持异步处理和消息队列
