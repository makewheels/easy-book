# Easy Book 数据库设计规范

## 核心设计原则

### 🚫 禁止使用数据库唯一约束

本项目严格遵守以下设计原则：
- **不在数据库层面使用唯一性约束（unique indexes）**
- **所有唯一性检查由业务逻辑层处理**
- **数据库索引仅用于查询性能优化**

## 设计理念

### 1. 为什么不使用数据库唯一约束？

#### ✅ 优势
1. **灵活性**：业务规则变更时无需修改数据库结构
2. **可控性**：可以在业务层实现更复杂的验证逻辑
3. **可维护性**：错误消息和验证逻辑集中管理
4. **可扩展性**：支持更灵活的冲突处理策略
5. **可测试性**：业务逻辑更容易进行单元测试

#### ⚠️ 传统方式的缺点
1. **硬编码**：业务规则被固化在数据库结构中
2. **错误处理困难**：数据库约束错误消息难以定制
3. **扩展性差**：复杂业务规则难以用数据库约束表达
4. **维护复杂**：变更业务规则需要数据库迁移

### 2. 实施策略

#### 数据库层职责
- ✅ 创建查询性能优化索引
- ✅ 保证数据一致性和事务完整性
- ✅ 提供高效的数据存储和检索

#### 业务逻辑层职责
- ✅ 实现所有唯一性检查
- ✅ 处理业务规则验证
- ✅ 提供用户友好的错误消息
- ✅ 实现复杂的业务逻辑

## 实际应用

### 预约系统案例

#### 传统方式（已废弃）
```javascript
// ❌ 已废弃：数据库唯一约束
db.appointments.createIndex([
    {"student_id": 1, "appointment_date": 1, "time_slot": 1}
], {unique: true})
```

#### 当前方式（推荐）
```javascript
// ✅ 推荐：仅创建查询优化索引
db.appointments.create_index([("student_id", ASCENDING)])
db.appointments.create_index("appointment_date")
db.appointments.create_index([("appointment_date", ASCENDING), ("time_slot", ASCENDING)])
```

#### 业务逻辑检查示例
```python
# ✅ 推荐：在业务层实现重复检查
async def create(appointment_data: dict) -> AppointmentModel:
    # 可选：添加业务层重复检查
    # if await check_duplicate_appointment(...):
    #     raise ValueError("该时间段已有预约")

    # 直接创建预约
    appointment_id = await db.create_appointment(appointment_data)
    return AppointmentModel(**appointment_data)
```

## 索引策略

### 允许的索引类型
- ✅ **单字段索引**：提升查询性能
- ✅ **复合索引**：优化多字段查询
- ✅ **降序索引**：支持排序需求
- ❌ **唯一索引**：禁止使用

### 推荐索引配置

#### students 集合
```javascript
// 查询优化索引
db.students.create_index("name")                    // 姓名查询
db.students.create_index("create_time", -1)         // 按创建时间倒序
```

#### appointments 集合
```javascript
// 查询优化索引
db.appointments.create_index([("student_id", ASCENDING)])                    // 按学生查询
db.appointments.create_index("appointment_date")                               // 按日期查询
db.appointments.create_index([("appointment_date", ASCENDING), ("time_slot", ASCENDING)])  // 按日期和时间查询
```

#### attendances 集合
```javascript
// 查询优化索引
db.attendances.create_index([("student_id", ASCENDING)])        // 按学生查询
db.attendances.create_index("attendance_date", -1)              // 按日期倒序
db.attendances.create_index("appointment_id")                   // 按预约查询
```

## 部署和运维

### 新项目初始化
1. 确保数据库配置文件中不包含 `unique=True`
2. 运行索引创建脚本时只创建性能优化索引
3. 在业务逻辑中实现必要的唯一性检查

### 现有项目迁移
1. 使用脚本删除现有的唯一索引
2. 更新应用代码，在业务逻辑中添加重复检查
3. 测试验证业务逻辑正确性

### 监控和维护
1. 定期检查查询性能，优化索引策略
2. 监控业务逻辑中的重复检查效果
3. 避免在数据库中创建新的唯一约束

## 代码审查检查点

在代码审查时，需要检查以下方面：

### 数据库层面
- [ ] 确保没有使用 `unique=True` 创建索引
- [ ] 索引仅用于查询性能优化
- [ ] 数据库配置文件符合规范

### 业务逻辑层面
- [ ] 必要的重复检查已实现
- [ ] 错误消息用户友好
- [ ] 业务逻辑经过充分测试

## 总结

这个设计规范体现了现代应用开发的最佳实践：
- **关注点分离**：数据库负责性能，业务逻辑负责规则
- **灵活性优先**：易于修改和扩展业务规则
- **用户体验**：提供更好的错误处理和消息提示
- **可维护性**：降低系统复杂度，便于团队协作

所有新开发的项目都应严格遵守此规范，确保系统的一致性和可维护性。
